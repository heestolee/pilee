# Frontend UI Engineering

## Overview

Build interfaces that are accessible, performant, and visually polished. The goal is UI that looks like it was designed and built by a skilled team — not generated by a model. This means following the project's design system, handling every state (loading, error, empty, success), supporting keyboard and screen reader access, and avoiding the recognizable "AI aesthetic."

## When to Use

- Building new components, pages, or screens
- Modifying existing user-facing interfaces
- Implementing responsive or adaptive layouts
- Adding interactivity, animations, or state management
- Fixing visual, layout, or UX bugs

## Component Structure

### Organization

Group related files together:

```
components/
  task-list/
    TaskList.tsx           → main component
    TaskList.test.tsx      → tests
    TaskListItem.tsx       → sub-component if needed
    useTaskList.ts         → local hook or state logic
```

### Design Guidelines

**Prefer composition over configuration.** Build small focused components and combine them, rather than one mega-component with many mode flags.

**Keep components focused.** A component does one job: renders a concept, exposes a small prop surface, delegates unrelated logic elsewhere.

**Separate data from presentation.** A container acquires data and handles loading/error/empty states. A presentational component renders the valid-state UI.

## State Management

Choose the simplest approach that works:

```
Component-local state     → UI toggles, form input, transient selections
Lifted state              → shared between siblings or nearby components
Context / provider        → theme, locale, session, permissions
URL state                 → filters, pagination, shareable view configuration
Server state              → fetched data with caching, revalidation, optimistic updates
Global store              → complex cross-cutting state (rare — prefer other options first)
```

Avoid prop drilling through many layers. If a value passes through components that do not use it, restructure the hierarchy or introduce a context boundary.

## Avoiding the AI Aesthetic

AI-generated UI has recognizable anti-patterns. Avoid all of them:

| AI Default | Production Quality |
|---|---|
| Purple/indigo color scheme | Use the project's actual palette |
| Excessive gradients | Flat or subtle, matching the design system |
| Maximum border-radius on everything | Consistent radius from the design system's scale |
| Generic hero sections | Content-driven layouts |
| Placeholder text that hides layout issues | Realistic content that tests real-world lengths |
| Uniform oversized padding | Consistent spacing from the design system's scale |
| Card grids everywhere | Purpose-driven layouts based on information priority |
| Heavy drop shadows | Subtle or none, per the design system |

## Spacing and Layout

Use a consistent spacing scale. Do not invent arbitrary values.

```
Use the project's scale (e.g., 4px increments):
  4px, 8px, 12px, 16px, 24px, 32px, 48px, 64px

Good:  padding: 16px; gap: 12px;
Bad:   padding: 13px; margin-top: 37px;
```

## Typography Hierarchy

```
h1 → page title (one per page)
h2 → section title
h3 → subsection title
body → default text
small → secondary or helper text
```

Do not skip heading levels. Do not use heading styles for non-heading content.

## Color

- Use semantic color tokens from the design system, not raw hex values
- Ensure 4.5:1 contrast for normal text, 3:1 for large text
- Never rely solely on color to convey meaning — pair with icons, text, or patterns

## Accessibility (WCAG 2.1 AA)

### Keyboard Access

Every interactive element must be reachable and operable by keyboard. Prefer native HTML elements. For custom controls, provide:
- correct semantic role
- keyboard activation behavior
- visible focus indicator
- state announcements where needed

### Labels

Every interactive element needs an accessible name. Associate labels, help text, and error messages with their controls.

### Focus Management

Move focus intentionally when content changes:
- dialog opens → focus enters the dialog
- dialog closes → focus returns to the trigger
- validation fails → focus or announcement guides the user

### Empty, Loading, and Error States

Never show a blank screen. For every state, explain:
- what happened
- whether the user should act
- what the next useful action is

## Responsive Design

Start with the smallest supported viewport and add layout complexity as space allows.

```
Verify at representative breakpoints:
  320px  → small phone
  768px  → tablet
  1024px → laptop
  1440px → desktop
```

## Loading and Transitions

- Use loading states that preserve layout expectations (skeleton screens, spinners in place)
- For optimistic updates, ensure a clear rollback path on failure
- Transitions should clarify state changes, not distract from content

## Common Rationalizations

| Rationalization | Reality |
|---|---|
| "Accessibility is a nice-to-have" | It is a legal requirement in many jurisdictions and an engineering quality standard. |
| "We will make it responsive later" | Retrofitting responsive design costs three times more than building it from the start. |
| "This is just a prototype" | Prototypes become production code. Build the foundation correctly. |
| "The AI aesthetic is fine for now" | It signals low quality. Use the project's design system from the start. |
| "The design is not final yet" | Use design system defaults. Unstyled UI gives reviewers a broken first impression. |

## Red Flags

- Components grown beyond a single coherent responsibility
- Inline styles or arbitrary pixel values outside the spacing scale
- Missing error, loading, or empty states
- No keyboard navigation testing
- Color as the only indicator of state (no text or icons)
- Generic AI look (purple gradients, oversized cards, stock layouts)

## Verification

After building UI:

- [ ] Component renders without console errors
- [ ] All interactive elements are keyboard accessible
- [ ] Screen reader can navigate the content structure
- [ ] Responsive: works at 320px, 768px, 1024px, 1440px
- [ ] Loading, error, and empty states are all handled
- [ ] Follows the project's design system (spacing, colors, typography)
- [ ] No accessibility violations found
